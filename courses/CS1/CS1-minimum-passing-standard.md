# CS1 Minimum passing standard

**Purpose:** What “passing” looks like for **leaders** and **students**—the **Evidence of Mastery** bar per module, plus the **final** bar. Detailed tasks and tracks: [03-syllabus-schedule/cs1-course-structure-overview.md](./03-syllabus-schedule/cs1-course-structure-overview.md).

This is a **minimum** bar. Your teacher may require more for a higher grade.

---

## Module 1 — GitHub Foundations

- Student can **clone, commit, push, pull, branch, and open a PR**
- **Commit messages are meaningful**
- Student can **explain** the basic collaboration workflow (in speech or short writing)

---

## Module 2 — HCI and Figma

- Student can identify **user needs** and **usability issues**
- Student can create a **prototype in Figma**
- Student can **explain design decisions** clearly
- Student can identify **possible bias** in questions, assumptions, or design choices
- **Track B only:** student can show **before → after** intent (current UI vs Figma) and justify **why** changes improve usability

---

## Module 3 — LaTeX

- Student can produce a **clean, readable** one-page LaTeX document
- Document has **clear structure** and formatting
- Student can use LaTeX to communicate **technical ideas**, not only plain text

---

## Module 4 — Login page (static)

- Login page **structure** is complete and semantic where it matters
- Layout is **readable** on **phone- and desktop-width** viewports
- **Bootstrap** is used appropriately (not only custom CSS)
- Login form contains **title**, **username or email**, **password**, **button**

---

## Module 5 — Login page (grid + flex)

- **Rows and columns** are used correctly on the login page
- **Flexbox** is used appropriately (centering, distribution, or component alignment)
- The **same** login page stays **login-centric** and responsive (not a separate multi-page marketing site)

---

## Module 6 — Dice Challenge

- Student can **update the DOM** in response to user actions
- **Event handling** works without breaking the page on repeated clicks
- **State** (last roll, sum, dice count, etc.) is **visible** in the UI
- Student can **explain briefly** how random values map to DOM updates  
- **Project minimum** (see overview Module 6): at least **one** die **1–6**, **vanilla JS only**, plus **at least two** features from the listed set (multi-dice, sum, reset, validated count, history, disabled-while-rolling)

---

## Module 7 — Next.js

- Student can build and organize **multiple frontend pages**
- Student uses **components** appropriately
- Student can connect frontend pages to **API data**

---

## Module 8 — FastAPI

- Student can **create, read, update, and delete** book data via the API
- **API routes work** as specified
- Student can **explain the purpose** of each main endpoint

---

## Module 9 — PostgreSQL

- Data is stored **persistently** (survives restart)
- Student can explain the **basic data model**
- Student understands how **endpoints connect** to the database

---

## Module 10 — Final (course pass)

To **pass the course**, the student must **meet Module 10 evidence** and submit the final artifact:

- Student can build a **complete basic full-stack** Book application
- Student can **explain** design and technical decisions
- Student can **demonstrate** a working path from **interface to database**
- Final deliverable includes (as one integrated project): **HCI input**, **Figma**, **Bootstrap login** (grid/flex), **JS-level interactivity in the Book app (via Next)**, **Next.js** frontend, **FastAPI** backend, **PostgreSQL**, **GitHub workflow**
